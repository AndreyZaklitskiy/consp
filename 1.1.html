Хуки

Суть хуков заключается в том что система делится на части, у каждой части есть своё название и есть определенные функции которые по цепочке запускаются и строят эту часть системы. И вот строятся одна за другой части системы. На пример. Одна часть занимается регистрацией типов записей, таксономий. Другая часть занимается регистрацией виджетов, созданием виджетов. Какая-то часть занимается выводом информации: выводом меню, выводом контента шаблонов и т.д.
Знать это нужно для того что-бы если нужно внедрить что-то своё то нужно знать в какой момент WordPress Строит ту или иную свою часть, подписаться на этот момент времени и передать какую-то свою ф-ию.

По сути в ВП если мы хотим добавить что-то своё то нужно знать хуки ВП и в нужный хук передать свою функцию внутри которой будут переданы те ф-ии которые есть в ВП.

Порядок выполнения хуков, какие хуки есть - это всё есть на любом из сайтов выше.

Работает так что в тех местах где нужно в шаблоне вставляются ф-ии php. Интересно то что на те моменты когда уже выводится контент можно подписаться тоже. Но тогда уже работают не do_action, работают фильтры. Фильтры это тоже разновидность хуков.(add_filter) Фильтр - это тоже цепочка ф-ий которые запускаются одна за одной. При этом ф-ия должна принимать какой-то параметр и что-то возвращать. Это может быть полезно когда надо модифицировать в-ку которую выводит ВП. Можно получить ту вёрстку которую вставляет ВП и видоизменить её на ту которую нам надо.

Через Composer

Composer - это пакетный менеджер для приложений на php. Похоже на npm.

Сохраняет зависимости в Composer.json, устанавливает зависимости (тот же WordPress) в одну строчку.
Потом кто-то другой что-бы развернуть этот проект ему будет достаточно 1 файла Composer.json вместо того что-бы искать все зависимости проекта.
Так же можно создать шаблон для себя что-бы использовать его потом в других проектах.

После установки Composer В нужной директории в консоли нужно вызвать команду сomposer.
После чего нужно создать файл composer.json (по идее он создаться сам либо уже либо после установки 1-й зависимости.) и написать в нем пустые фигурные скобки.
composer require wordpress, johnbloch/wordpress

Создать БД, войти.

На странице всё выводится последовательно, в каждый скрипт ВП проваливается и только закончив его обрабатывать/выводить, он идёт выводить дальнейший html.

Далее создать все необходимые шаблоны: - Главная страница
В качестве главной стр. у нас установлена статическая страница (главная). В таком случае для неё будет создан специальный шаблон.

- Страница с блогом
Дело в том что когда мы выбрали для главной и для блога статическую страницу то WordPress автоматически настраивает страницу с последними записями на шаблон index.php. Тоесть Блог будет выводиться через index.php. Так же мы знаем что index.php - это особенный файл который работает в тех случаях когда не найден подходящий шаблон. Так что его надо настроить таким образом что-бы он мог выводить как последние записи так и любые другие записи с какой-то вёрсткой. Тоесть эту страничку надо будет настроить универсально. По умолчанию WordPress будет использовать index.php в качестве страницы Блога.

Вывод записи в WordPress

К тому моменту как wordpress приступит к сборке страницы у него уже, после построения своей системы, будут готовы объекты, переменные и массивы в которых хранится контент необходимый для выведения на этой странице. Нам осталось только организовать этот вывод.

ЦИКЛ wordpress. Стандартный цикл ВП подразумевает что внутри шаблона ВП сам понимает какой контент нужно вывести. Тоесть при вызове ВП знает какую именно запись запросил пользователь. При чем не важно предусмотрена 1 запись или много в любом случае используется цикл.

Если есть записи то работаем с секцией секцией:

the_posts_pagination(); - добавляет пагинацию.

Пагинацию лучше добавлять с помощью the_post_pagination();. Дело в том что это очень сложный элемент, по этому лучше сделать вёрстку в соответствии с ВП. или переделать имеющуиеся стили пагинации под ВП в имеющейся вёрстке.
